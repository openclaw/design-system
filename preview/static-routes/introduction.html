Introduction

One visual contract for the OpenClaw product family.

Carapace is the shared design system for OpenClaw applications, operational tools, public sites, documentation, and agent experiences. It gives those products one durable visual language without pretending that every surface has the same job.

Purpose

Consistency without flattening every product.

OpenClaw spans a menu bar app, full application panes, browser controls, documentation, public sites, and conversational workflows. Carapace supplies the decisions that should repeat across them: semantic color, typography, spacing, focus, motion, control density, data hierarchy, agent activity, and application anatomy. The result is a common grammar rather than identical pixels. Native utilities keep native materials; editorial pages keep a comfortable reading rhythm; both remain recognizably OpenClaw.

Extracted from working products, not invented in isolation.

The initial contract came from OpenClaw's public site, ClawHub, and the documentation surface. Application patterns face a higher bar: repeated structures in the macOS app and Control UI must share useful anatomy without importing consumer policy. Carapace records proven decisions and evaluates new ones; it is not a speculative component warehouse.

Ownership

Share visual anatomy. Keep product behavior local.

Carapace owns framework-neutral tokens, themes, base behavior, component roles, and composition rules. It can define sidebar anatomy, provider and model hierarchy, streaming activity, or the relationship between a pane header, body, and footer. Consumers keep routing, data, authorization, persistence, native window management, and business rules. The model picker defines grouping and interaction; the app decides which models are installed, permitted, and stored.

The shared contract is intentionally small.

Stable selectors describe roles that survive implementation changes: surface, control, status, pane, and composer. Shared roles belong here. Specialized workflows stay in the owning product and compose those roles.

The boundary is part of the design.

The macOS app keeps SwiftUI and AppKit semantics, commands, sheets, materials, and window sizing. The Control UI keeps Lit views, WebSocket state, browser navigation, responsive drawers, and local persistence. Both preserve the same hierarchy, control roles, status language, and responsive intent.

Consumers

One family, several operating contexts.

macOS app

Settings, Quick Chat, model selection, panes, and menu bar utilities. Carapace informs hierarchy and density while the app preserves native interaction.

Control UI

Browser settings, sessions, agents, channels, navigation, chat controls, and inspectors composed inside existing application state and routing.

Public and editorial sites

OpenClaw.ai, organization pages, releases, and documentation use the same tokens with a calmer reading measure.

Agent experiences

Messages, tools, questions, attachments, multi-agent coordination, progress, and errors need clear state without noisy cards.

These contexts are evidence sources as well as consumers. Every proposal should name the real surfaces, shared invariant, and local adaptations instead of hiding differences behind a generic mockup.

Architecture

Adopt from the foundation upward.

Consumers can take the smallest contract that solves the problem. A complete import suits a new surface; focused entry points make ownership and maturity explicit in established products. Later layers build on semantic foundations rather than replacing them with route-specific values.

Import order communicates intent

Load tokens and themes first, then typography, base behavior, stable components, and explicitly chosen candidate layers. Adapters are transitional, not a second source of truth.

Maturity

Stable, Candidate, and Lab mean different promises.

Stable contracts are compatibility-sensitive package surfaces, tested against current consumers and released under immutable semantic tags. Candidate contracts are useful and importable but remain opt-in until multiple products validate them. Lab examples make an idea inspectable before it becomes package API. Promotion requires repeated need, clear ownership, accessible behavior, and a durable selector contract, not merely a polished screenshot.

Application parity

Parity is shared intent, not pixel duplication.

The macOS app and Control UI should agree on information order: global and local navigation, model and reasoning controls, agent activity, inspectors, and status. Operational surfaces favor compact rows, restrained borders, predictable navigation, and immediate access to detail. The primary task owns most of the viewport while rails collapse before they squeeze the work area.

Motion should explain disclosure, streaming, progress, or entry. Continuous decoration, layout-shifting transitions, and effects that obscure state are not parity. Every animated contract includes reduced-motion behavior and remains understandable in a static frame.

Definition of parity

Parity includes information hierarchy, states, keyboard behavior, density, motion intent, responsive strategy, and ownership boundary, not merely similar colors.

Preview workflow

Use the reference as an inspection tool.

The Home grid is an inventory, not the specification. Open a component page for its specimen, variants, states, responsive canvas, guidance, and markup. Exercise application and agent pages like compact products: change theme and viewport, open controls, test empty and error states, and grow labels.

  1. Start with the role.Choose the component or composition that matches the job, not the closest decoration.
  2. Inspect every state.Check focus, hover, selection, loading, streaming, failure, disabled, empty, and reduced-motion behavior where applicable.
  3. Read Usage before Code.The ownership and composition guidance matters as much as the example markup.
  4. Adapt inside the consumer.Use local framework primitives and product state while preserving the shared hierarchy and semantic roles.
  5. Bring evidence back.If a selector needs consumer-specific exceptions, document the mismatch before promoting or widening the contract.

Workbench controls, route shells, fixture data, and demo bindings are preview scaffolding, not package API. Consumers use the documented anatomy or CSS entry point, not the preview runtime.

Accessibility

Accessible behavior is part of the visual contract.

Interactive examples need visible focus, accurate names, a keyboard path, and semantics that survive without color or animation. Controls reserve room for labels and loading states so interaction does not shift layout. Consumers test both themes, zoom, long labels, narrow viewports, reduced motion, and assistive navigation. Native products retain native semantics; web products start with real buttons, links, inputs, dialogs, and landmarks.

Adoption

Use the smallest layer that solves the problem.

Install an immutable release tag, then import the complete contract or focused entry points for tokens, themes, typography, base styles, components, and approved candidates. Existing products should map local values to semantic tokens first, remove duplicate base behavior, and replace one repeated role at a time. Skills follow the default branch for current guidance; runtime assets remain pinned to releases.

Contributing

Promote evidence, not one-off preference.

A component belongs here when multiple consumers need the same interface or an existing stable role is incomplete. Identify the real surfaces, shared invariant, and local differences before coding. Changes include focused tests, reference guidance, responsive and theme evidence, and a maturity decision. Product-specific workflows that do not generalize should remain local.
