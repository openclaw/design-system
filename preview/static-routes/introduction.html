Introduction

One visual contract for the OpenClaw product family.

Carapace is the shared design system for OpenClaw applications, operational tools, public sites, and agent experiences. It defines the visual language and reusable anatomy; each product keeps ownership of its routes, data, permissions, and behavior.

Purpose

Consistency without flattening every product.

OpenClaw spans a menu bar app, full application panes, control surfaces, documentation, and conversational workflows. Those products should feel related without pretending they have identical jobs. Carapace supplies stable decisions for color, type, spacing, focus, motion, controls, data density, and agent activity so teams can spend their energy on product behavior.

The system deliberately avoids owning business logic. A model picker can define structure, state, and interaction expectations while the consuming application still decides which providers are available, how a session override is stored, and whether an option is permitted.

Principles

Quiet, compact, legible, and explicit.

Product first

Operational screens favor scanability and repeated action over decorative composition.

Dense, not cramped

Controls stay compact while text, focus, touch targets, and hierarchy remain clear.

State is visible

Loading, selection, progress, failure, permissions, and streaming never rely on color alone.

Motion explains

Animation connects state changes. Reduced-motion behavior is part of the same contract.

Consumers retain ownership

Carapace provides anatomy and visual behavior, not application routing or domain policy.

Examples are executable

Reference pages expose live states, responsive canvases, usage guidance, and canonical markup.

Adoption

Use the smallest layer that solves the problem.

Import the package styles for a complete surface, or adopt the documented candidate layers while a contract is being evaluated. Start with tokens and base styles, then choose components by role. Do not copy preview-only scaffolding into products; the reference shell exists to inspect states, themes, and responsive behavior.

Definition of parity

A Carapace component is not “done” because its colors look similar. It must carry the same information hierarchy, states, keyboard behavior, density, motion intent, and ownership boundary as the product surface it represents.
